    为何这么复杂

           日常工作中，有时为了查Issue，或者为了维护现有项目，我们常常需要阅读老代码，来理顺一段业务逻辑。这通常都是一个痛苦的经历。那些如迷宫一般的代码，一旦陷进去，没个大半天就别想绕出来。大量的时间耗费其中。甚至很多老项目，由于当时开发人员的离开，再也没有人能搞明白那个项目是干嘛的。这样的情形下，我们交付的产品质量自然谈不上多高。UI反人类、性能低都是小事情，更关键的是我们没有办法正确而快速地响应用户的需求。我常常会感叹“怎么会这么复杂？”。

           对于这个问题，一个直白的答案是“因为业务本来就这么复杂”。这当然没错。我们业务系统面向现实的业务逻辑，而业务逻辑的复杂性是无上限的。正如《企业应用架构》中所说：“没什么比业务逻辑更没有逻辑”。然而，这是我们写出糟糕代码、交付低质量产品的理由吗？当然不是。因为我们的职责，就是开发出软件来应对业务复杂性。Hold住复杂性，本身就是我们的责任。而这也是企业级应用开发的核心难点与魅力。

    事务脚本、贫血模型与失忆症

           那么，到底是为什么我们的代码那么复杂、以至于我们Hold不住业务复杂性呢？我认为，是由于开发过程中，我们是在“面向技术实现编程”，而不是在“面向业务领域”进行编程。在开发过程中，我们关注的焦点不是“怎么用代码清晰描述出业务逻辑”，而是习惯于一头扎进技术细节，急不可耐地想要把乱七八糟的字段写到数据库中。只要最终凑出来的数据对了，就万事大吉。也许当时是对了，但是由于代码中几乎不包含业务知识，对于后期维护来说就是一场噩梦。

           “面向技术实现编程”的具体体现就是事务脚本和贫血模型。所谓事务脚本，是指过程化、命令式的编程模式。它将单个业务逻辑组织成一个过程，在此过程中根据需要修改业务数据，最后存入数据库。所谓贫血模型，是指只含有属性、不包含方法的类。这些类只是数据库中的数据的载体，本身没有任何行为。

    优点

    • 简单明了，没有难以理解的概念。
    • 性能好

    缺点

    • 以数据为核心，没有反映领域逻辑，造成领域知识丢失。
    • 三层架构模式，形成了一个庞大而复杂的Business层。职责混淆。
    • “面向过程”的编程方式，阻碍了“面向对象”各种模式的实施。代码极度缺乏良好的组织。

           最终的结果就是，随着业务规模的扩大，软件本身的复杂度呈现指数增长。

    复杂性的应对之道——领域驱动设计

    定义

    以领域模型为核心，而非以数据为核心的开发模式。

    工作流程

    1. 以一种领域专家、设计人员、开发人员都能理解的通用语言作为相互交流的工具，在交流的过程中发现领域概念，然后将这些概念设计成一个领域模型。（知识消化与建模）
    2. 领域模型驱动软件设计，用代码来实现该领域模型。（实现领域模型）
    3. 以实现的领域模型为核心，构建应用。

    为什么需要领域驱动设计

    业务价值：

    • 使领域专家和开发者一起工作，这样开发出来的软件能够准确地传达业务规则。
    • 将知识集中，确保知识不只是掌握在少数人手中。

    开发价值：

    • 贯彻面向对象编程理念，设计模式终于有了用武之地。
    • 更好的应用架构。
    • 清晰的边界，与微服务相得益彰。
    • 提升开发效率。

    实施领域驱动设计的挑战

    1. 需要理解大量新的概念，有一定学习成本。
    2. 开发人员对技术感兴趣，天生排斥领域知识。
    3. 建模耗费大量的时间。
    4. 项目初期工作量大。


    领域模型

    定义

    领域模型是对领域知识的组织和抽象。

    形式

    • 图形、文字形成的文档。
    • 日常沟通语言。
    • 代码。

    作用

    • 模型是团队所有成员使用的通用语言的中枢。
    • 模型是浓缩的知识。
    • 模型与实现紧密关联。

    理解领域模型的关键

    1. 忘记数据库！不要考虑要怎么将数据存到数据库中。
    2. 忘记客户端！不要考虑怎么处理请求，怎么返回响应。

    数据持久化和应用逻辑都不属于领域模型的范畴。

    有效建模的要素

    • 模型和实现绑定。
    • 建立一种基于模型的通用语言。
    • 开发一个蕴含丰富知识的模型。
    • 持续提炼模型。


    战略建模

    问题空间与解决方案空间

    问题空间：业务所面临的挑战。

    解决方案控件：如何实现软件以解决这些业务挑战。

    子域和限界上下文

    待补充

    PO领域子域划分


    上下文映射

    • 合作关系（Partnership）：两个上下文紧密合作的关系，一荣俱荣，一损俱损。
    • 共享内核（Shared Kernel）：两个上下文依赖部分共享的模型。
    • 客户方-供应方开发（Customer-Supplier Development）：上下文之间有组织的上下游依赖。
    • 遵奉者（Conformist）：下游上下文只能盲目依赖上游上下文。
    • 防腐层（Anticorruption Layer）：一个上下文通过一些适配和转换与另一个上下文交互。
    • 开放主机服务（Open Host Service）：定义一种协议来让其他上下文来对本上下文进行访问。
    • 发布语言（Published Language）：通常与OHS一起使用，用于定义开放主机的协议。
    • 大泥球（Big Ball of Mud）：混杂在一起的上下文关系，边界不清晰。
    • 另谋他路（SeparateWay）：两个完全没有任何联系的上下文。


    战术建模

    实体 Entity

    实体就是领域中需要唯一标识的对象。

    特征：

    1. 具有唯一标识。
    2. 具有生命周期。
    3. 是可变的。

    值对象 Value Object

    用于描述领域的某个方面而本身没有概念标识的对象成为值对象。

    特性：

    1. 是不可变的。
    2. 通知值来标识相等性，而非某个标识。

    领域服务 Domain Service

    当某些重要的过程或转换操作不是实体或值对象的职责时，就需要创建一个领域服务来表示它。领域服务是无状态的。

    聚合 Aggregate

    聚合就是一组相关对象的集合，我们把它作为修改数据的单元。每个聚合都有一个根（AggregateRoot）和一个边界。边界定义了聚合内部都有什么。根则是聚合所包含的一个特定的实体。对聚合而言，聚合外部的对象只可以引用根，聚合内部的对象之间则可以互相引用。

    聚合定义了一个事务边界。在聚合内部的，所有对象必须永远保持一致性，即必须满足业务规则，也就是强一致性。在聚合外部，则不要求不同聚合随时保持一致。


    例：

    要求：在并发条件下，让PO总是满足业务规则。

    解决方案：创建聚合。

    工厂 Factory

    提供复杂对象的创建服务。

    资源库（仓储） Repository

    资源库提供一种“内存集合”的抽象概念。它表示“已经存在的对象”。通常我们只为聚合根提资源库。

    当我们想要获取一个已经存在的对象时，需要通过资源库获取引用。

    当我们新建一个对象，且希望“永久保留”该对象时，需要将其添加进资源库。

    在具体实现上，资源库封装了数据库调用。


    DDD应用架构

    六边形架构



           领域层位于最里层，或者称为最底层。领域层对应领域模型，它是一个运行于内存中的对象模型，通过各种DDD构件，和和数据库、外部API、应用逻辑彻底解耦。

    CQRS 读写分离模式

    实际项目结构

    展示层 Presentations

    提供对外API接口。我们的项目中还提供位于顶层的全局依赖注入器。

    应用层 Application

    定义软件要完成的任务，并且指挥表达领域概念的对象来解决问题。

    应用层要尽量简单，不包含业务规则或者知识，而只是为下一层中的领域对象协调任务，分配工作，使它们互相协作。它告诉领域层要做什么，但是怎么做由领域层负责。

    领域层 Domain

    领域模型的具体实现。它负责表达业务概念，业务状态信息以及业务规则。位于整个应用结构的最中心位置。

    基础设施层 Infrastructure

    实现各种领域层定义的外部接口，封装外部依赖。主要是对数据库和外部API的访问。

    公共组件层 Common

    各层公用


    仓储的实现

    Query——ReadOnlyRepository

    核心目的：解决查询的多样性和效率问题。

    解决方案：选择轻量级ORM工具——Pikaq。

    Command——Repository

    核心目的：对抗"对象-关系"阻抗适配。

    解决方案：选择重量级ORM工具——Entity Framework Core。

    工作单元模式（UnitOfWork）

    追踪对象状态的变化，并在合适的时机将修改批量写入数据库，并提供事务功能。
